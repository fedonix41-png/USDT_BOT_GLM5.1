# Техническое задание: Telegram-бот для обмена USDT

## 1. Общая информация
**Цель:** Автоматизация приёма заявок на покупку и продажу USDT с гибкой системой ролей и уведомлений.  
**Стек:** Python 3.11+, Aiogram 3.x, FastAPI (вебхук), PostgreSQL 15, Redis, ARQ (фоновая очередь), Docker.  
**Роли:** SuperAdmin, Admin, Operator, Client.

## 3. Пользовательские сценарии и экраны

### 3.1. Регистрация и первый запуск
При старте `/start` бот сохраняет пользователя в БД с ролью `client`. Приветственное сообщение и меню в зависимости от роли.

### 3.2. Клиент: Покупка / Продажа USDT

**FSM `OrderCreation`:**
- **State `waiting_amount`**  
  **Сообщение:**  
  *Введите сумму в USDT, которую хотите купить.*  
  (или *продать*)  
  **Валидация:** положительное число, максимум 100000.  
  **Действие:** сохранить сумму, перейти к подтверждению.

- **State `confirm_order`**  
  **Сообщение:**  
  ```
  📌 Заявка на покупку USDT
  Сумма: {amount} USDT
  Курс: {rate} руб/USDT
  К оплате: {total_fiat} RUB
  Реквизиты: {payment_link}

  После оплаты ожидайте подтверждения.
  ```
  **Inline-кнопки:**  
  `[🔗 Ссылка не работает]` `[❌ Отменить заявку]`

  **Алгоритм:**
  1. Получаем текущий курс покупки и флаг `buy_enabled` из `GlobalSettings`.
  2. Рассчитываем сумму в фиате.
  3. Извлекаем `payment_link_buy` (расшифровываем).
  4. Создаём запись `Order` со статусом `created`.
  5. Отправляем сообщение с клавиатурой.
  6. Отправляем уведомление в настроенные чаты о новой заявке.

**Обработка кнопок:**
- *«Ссылка не работает»* → бот отвечает: *«Мы уже меняем ссылку. Новая ссылка будет отправлена сюда же.»* Отправляет алерт в чаты уведомлений с ID заявки и пользователя. Админ, сменив глобальную ссылку, инициирует замену: все новые заявки получают новую ссылку, а клиенту в это сообщение редактируется реквизит + уведомление *«Ссылка обновлена. Актуальные реквизиты выше.»*.
- *«Отменить заявку»* → меняет статус на `cancelled`, редактирует сообщение: *«❌ Заявка отменена.»*

### 3.3. Оператор: Активные заявки, Статистика

- **«Активные заявки»**: вывод списка всех `Order` со статусом `created` за последние 24 часа. Пагинация: по 5 заявок, кнопки `◀️ Назад` `Вперёд ▶️`.  
  Формат сообщения:  
  ```
  Заявка #{id} | {buy/sell}
  Сумма: {amount} USDT | {status}
  Дата: {created_at}
  ```
- **«Статистика за период»**: FSM из двух состояний (ввод даты начала, даты окончания). Вывод:  
  ```
  📊 Статистика с {start} по {end}:
  Покупок: {count_buy} на сумму {sum_buy} USDT
  Продаж: {count_sell} на сумму {sum_sell} USDT
  Всего заявок: {total}
  ```

### 3.4. Администратор: изменение курсов и реквизитов

- **«Сменить курс (покупка)»**: запрос нового курса. Сохраняется в `Rate` и кеш Redis. Сообщение: *«Курс покупки изменён на {new_rate} RUB/USDT»*. Логируется в аудит.
- **«Сменить реквизиты»**: FSM – выбор типа (покупка/продажа), ввод новой ссылки/реквизитов. Сохраняется зашифрованным в `GlobalSettings`. Сообщение: *«Реквизиты для {покупки|продажи} обновлены.»* Если есть клиенты с «битой» ссылкой, им автоматически рассылается обновление в их последнее сообщение с заявкой.

- **«Стоп закуп / Стоп продажа»**: инвертирует флаг. Сообщение: *«Покупка USDT остановлена.»* / *«…возобновлена.»*
- **«Отключить бота»**: глобальный флаг `bot_enabled`. Пока выключен, все клиентские команды отвечают: *«Бот временно недоступен.»* Админы могут работать.

- **«Чаты уведомлений»**: подменю  
  ```
  [📋 Список чатов] [➕ Добавить чат] [➖ Удалить чат]
  ```
  Добавление: переслать боту сообщение из нужного чата (или просто ввести ID). Бот проверяет, что он админ в чате, и сохраняет.
- **«Сделать Оператором»**: запрос Telegram ID или пересылка контакта. Роль обновляется.

### 3.5. SuperAdmin: дополнительно «Сделать Админом»

Аналогично, но с ролью `admin`. Команда только у SuperAdmin.

## 4. Алгоритмы ключевых процессов

### 4.1. Управление состоянием бота
Флаги хранятся в таблице `global_settings` (ключ-значение):
- `bot_enabled` (0/1)
- `buy_enabled` (0/1)
- `sell_enabled` (0/1)

Проверка происходит в middleware перед обработкой клиентских действий. При значении 0 возвращается информационное сообщение.

### 4.2. Управление ссылками и шифрование
Реквизиты:
- `payment_link_buy`
- `payment_link_sell`

Шифрование AES-256 CBC с ключом из переменных окружения `ENCRYPTION_KEY`.  
При записи значение шифруется, при чтении расшифровывается.  
При смене ссылки админ вводит новое значение в открытом виде.  
При нажатии «Ссылка не работает» клиентский обработчик:
1. Устанавливает в заявке флаг `link_broken = True`.
2. Отправляет сообщение в чаты уведомлений: *«⚠️ Клиент @{username} жалуется на ссылку в заявке #{id}»*.
3. Когда админ меняет глобальную ссылку, запускается фоновая задача: найти все активные заявки с `link_broken = True`, для каждого клиента отредактировать его последнее сообщение с заявкой (бот хранит `message_id` и `chat_id` в заявке), заменив реквизиты и сбросив флаг.

### 4.3. Статистика
SQL-запрос агрегации по диапазону дат (`created_at BETWEEN :start AND :end`). Индексы по `created_at`, `status`, `order_type`.

### 4.4. Уведомления в чаты
При создании заявки, жалобе на ссылку, назначении ролей – событие публикуется в очередь (ARQ) с retry. Воркер берёт задачу, получает список чатов из БД и отправляет сообщения с обработкой ошибок Telegram API (retry 3 раза, затем лог ошибки).

## 7. Сценарии обработки ошибок

| Ситуация                                 | Поведение                                                                                             |
| ---------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Потеря соединения с Redis                | Бот переходит в режим работы без кеша (курсы — прямой запрос в БД, FSM хранить в памяти или отключить). Логирование ошибки, повторное подключение по таймеру. |
| Потеря соединения с PostgreSQL           | Бот должен уведомлять пользователя: *«Сервис временно недоступен, попробуйте позже.»*. Реализуется через `try/except` в репозиториях с возвратом ошибки и отправкой сообщения. |
| Ошибка отправки сообщения в Telegram     | Очередь (ARQ) автоматически повторяет до 3 раз. При исчерпании попыток логгирует `failed_notification` и удаляет чат из списка уведомлений, уведомляет админа. |
| Бот не является администратором в чате уведомлений | При добавлении чата проверяется `getChatMember`. Если нет прав — сообщение: *«Бот должен быть администратором чата.»* |
| Некорректный ввод (сумма, дата, ID)      | Валидация в FSM с просьбой ввести правильное значение. Максимум 3 попытки, затем выход.               |
| Неверный Telegram ID при назначении роли | Проверка существования пользователя в БД. Если нет — сообщение: *«Пользователь не найден.»*            |
| Одновременное изменение настроек         | Оптимистичные блокировки не требуются, достаточно атомарных операций обновления значения в БД.       |

## 8. Roadmap для AI-кодинга

Эта последовательность учитывает наращивание контекста, позволяя AI-ассистенту генерировать код с минимальными противоречиями.

### Этап 1: Инициализация проекта и инфраструктура
1. **Файлы**: `requirements.txt`, `Dockerfile`, `docker-compose.yml`, `.env.example`, `config.py`.
2. **Настройка логирования** (`utils/logger.py`).
3. **main.py**: подключение бота, FastAPI (вебхук), инициализация Redis, ARQ.

### Этап 2: База данных
1. **Модели SQLAlchemy** (все таблицы) в `src/models/`.
2. **Инициализация Alembic**, создание первой миграции.
3. **Базовый класс репозитория** с методами `create`, `get_by_id`, `update`.

### Этап 3: Репозитории и сервисы (базовые CRUD)
1. **UserRepo**, **UserService** (регистрация, назначение ролей).
2. **SettingsRepo**, **RateRepo**, **OrderRepo**.
3. **Encryption service** (шифрование/дешифрование реквизитов).

### Этап 4: Middleware
1. `RoleMiddleware` (проверка роли, инжектирование объекта пользователя).
2. `BotEnabledMiddleware` (глобальный флаг).
3. `ThrottlingMiddleware` (антиспам).

### Этап 5: Клавиатуры и общие хендлеры
1. Создание клавиатур для всех ролей.
2. Хендлер `/start` с определением роли и отправкой соответствующего меню.

### Этап 6: Хендлеры клиента
1. FSM покупки/продажи (состояния, валидация, создание заявки).
2. Обработчик inline-кнопок «Ссылка не работает», «Отменить заявку».

### Этап 7: Хендлеры оператора
1. Просмотр активных заявок с пагинацией.
2. Статистика за период (FSM).

### Этап 8: Хендлеры администратора
1. Смена курсов.
2. Управление реквизитами (включая шифрование).
3. Флаги «стоп закуп/продажа», «отключить бота».
4. Управление чатами уведомлений.
5. Назначение ролей.

### Этап 9: Система уведомлений
1. Реализация ARQ-задач (отправка в чаты, обработка жалоб на ссылку).
2. Интеграция вызова задач в сервисы заявок и ссылок.

### Этап 10: Обработка ошибок и отказоустойчивость
1. Декораторы повторных попыток для БД/Redis.
2. Глобальный обработчик исключений в aiogram.
3. Проверка прав бота в чатах.

### Этап 11: Тестирование и деплой
1. Базовые интеграционные тесты (опционально, если AI будет генерировать).
2. Инструкции по деплою.
