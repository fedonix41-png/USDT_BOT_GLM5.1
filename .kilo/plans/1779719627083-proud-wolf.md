# План: Web-админка (REST API)

## 1. Обзор

REST API для управления курсами, заявками, пользователями и настройками бота. Авторизация через JWT токены, ролевой контроль доступа, интеграция с существующими сервисами.

### Цели
- Предоставить HTTP API для внешних систем (мобильное приложение, веб-интерфейс)
- Переиспользовать существующие сервисы (UserService, OrderService, RateService, SettingsService)
- Обеспечить безопасность: JWT авторизация, ролевой контроль, rate limiting

### Технологии
- **Framework:** aiohttp (текущий этап) → FastAPI (продакшен с SSL)
- **Auth:** JWT (PyJWT) — access token 30 мин, refresh token 7 дней
- **Validation:** Pydantic v2
- **Docs:** OpenAPI/Swagger (с FastAPI)

## 3. Безопасность

### 3.1 JWT Авторизация

**Токены:**
- **Access Token:** RS256 (или HS256), expires 30 мин
- **Refresh Token:** Хранится в БД, expires 7 дней, одноразовый (ротация при использовании)

**Claims:**
```json
{
  "sub": "user_id:123",
  "role": "admin",
  "exp": 1234567890,
  "iat": 1234567890,
  "jti": "unique-token-id"
}
```

**Login flow:**
1. Клиент отправляет `POST /api/v1/auth/login` с `{"telegram_id": 123456, "api_key": "secret"}`
2. Сервер проверяет: пользователь существует + роль >= operator + api_key совпадает с API_SECRET_KEY
3. Генерирует access + refresh токены, сохраняет refresh в БД (таблица api_tokens)
4. Возвращает оба токена

**Refresh flow:**
1. Клиент отправляет `POST /api/v1/auth/refresh` с `{"refresh_token": "..."}`
2. Сервер проверяет валидность refresh токена в БД, удаляет старый
3. Генерирует новую пару токенов

### 3.2 Ролевой контроль

**Защищённые endpoints:**

| Endpoint | Минимальная роль | Описание |
|----------|------------------|----------|
| `GET /api/v1/users` | admin | Список пользователей |
| `PATCH /api/v1/users/{id}/role` | super_admin | Назначение админа |
| `GET /api/v1/orders` | operator | Список заявок |
| `PATCH /api/v1/orders/{id}/status` | operator | Изменение статуса |
| `POST /api/v1/rates` | admin | Установка курса |
| `PATCH /api/v1/settings` | admin | Изменение настроек |
| `GET /api/v1/statistics` | operator | Статистика |

### 3.3 Дополнительная защита

- **Rate Limiting:** 100 req/min per IP (Redis sliding window)
- **CORS:** Whitelist origins из конфига
- **IP Whitelist:** Для admin/super_admin endpoints — проверка IP из конфига
- **Input Validation:** Pydantic схемы для всех входящих данных
- **Error Handling:** Не раскрывать детали ошибок, логировать в audit_logs
- **HTTPS:** Обязательно в продакшене (за nginx/proxy)
- **Brute Force:** Блокировка login на 5 мин после 5 неудачных попыток (Redis)

## 4. Модели данных

### 4.1 Новая таблица: api_tokens

```sql
CREATE TABLE api_tokens (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash VARCHAR(64) NOT NULL,  -- SHA-256 от refresh токена
    jti VARCHAR(36) NOT NULL UNIQUE,   -- JWT ID
    expires_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    revoked BOOLEAN DEFAULT FALSE
);

CREATE INDEX ix_api_tokens_user_id ON api_tokens(user_id);
CREATE INDEX ix_api_tokens_jti ON api_tokens(jti);
```

**Миграция:** `migrations/versions/003_add_api_tokens.py`

## 8. Риски и меры

| Риск | Мера |
|------|------|
| JWT key compromise | Хранить в .env, ротация ключей |
| Rate limit bypass | Redis sliding window, блокировка IP |
| SQL injection | SQLAlchemy parameterized queries |
| XSS | JSON API без HTML, Content-Type: application/json |
| Brute force login | Redis счётчик попыток, блокировка на 5 мин после 5 попыток |
| Token theft | Короткий access token, HTTPS, Secure flag для cookies |
| Admin API from untrusted IP | IP whitelist для admin/super_admin endpoints |
| Credential stuffing | API key проверка, rate limit на login |
