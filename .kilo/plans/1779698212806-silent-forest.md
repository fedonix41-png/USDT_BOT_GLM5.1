## Структура нового docs/roadmap.md

### Принцип
Roadmap разделён на 4 приоритетных уровня:
- **P0 — Критично для production** (без этого нельзя деплоить)
- **P1 — Важно для стабильности** (снижает риск инцидентов)
- **P2 — Полезно для операционной эффективности** (улучшает UX/мониторинг)
- **P3 — Расширение функциональности** (новые фичи, дальняя перспектива)

### Содержание roadmap

#### P0 — Критично для production

1. **Graceful shutdown** (SIGTERM)
   - Источник: risks.md п.3, FULL_TZ неявно
   - Что: обработка SIGTERM в main.py → stop_polling, close sessions, dispose engine
   - Почему: без этого Docker `restart` убивает процесс посреди транзакции

2. **Глобальный обработчик ошибок aiogram**
   - Источник: FULL_TZ секция 7, Этап 10
   - Что: `@dp.errors` → логирование + сообщение пользователю «Произошла ошибка, попробуйте позже»
   - Почему: сейчас необработанные исключения молча ломают handler, пользователь не получает ответа

3. **Блокировка пользователей (is_blocked)**
   - Источник: FULL_TZ (поле есть в модели), анализ кода (поле не проверяется)
   - Что: проверка `user.is_blocked` в UserMiddleware → если True, сообщение «Вы заблокированы»
   - Почему: без этого нет механизма блокировки злоупотребляющих клиентов

#### P1 — Важно для стабильности

4. **Обработка потери соединения с БД**
   - Источник: FULL_TZ секция 7
   - Что: try/except в DBSessionMiddleware → при ошибке подключения сообщение «Сервис временно недоступен»
   - Почему: при падении PostgreSQL бот получает необработанное исключение на каждом запросе

5. **ThrottlingMiddleware (антиспам)**
   - Источник: FULL_TZ Этап 4 п.3
   - Что: ограничение 1 сообщение/сек на пользователя для команд, 5/мин для FSM-ввода
   - Почему: защита от спама и случайной массовой отправки

6. **Лимит попыток FSM-ввода**
   - Источник: FULL_TZ секция 7: «Максимум 3 попытки, затем выход»
   - Что: при невалидном вводе (сумма, дата, ID) — после 3 ошибок сброс FSM с сообщением
   - Почему: сейчас пользователь может бесконечно вводить неверные данные

#### P2 — Полезно для операционной эффективности

7. **Redis-кеш флагов вместо in-memory dict**
   - Источник: risks.md п.1, архитектура.md
   - Что: заменить `_bot_enabled_cache` dict на Redis с TTL 30 сек
   - Почему: при нескольких инстансах бота (горизонтальное масштабирование) in-memory кеш не синхронизируется

8. **Структурированное JSON-логирование**
   - Источник: risks.md п.2
   - Что: заменить logging.basicConfig на JSON-форматтер
   - Почему: в Docker удобнее парсить JSON-логи (Grafana Loki, ELK)

9. **Healthcheck HTTP-эндпоинт для bot-контейнера**
   - Источник: risks.md п.4
   - Что: aiohttp/web на порту 8080 → GET /health → 200 OK + проверка DB ping
   - Почему: Docker не может определить, жив ли бот (сейчас только restart: unless-stopped)

10. **Улучшение обработки уведомлений**
    - Источник: FULL_TZ секция 7: «При исчерпании попыток логгирует failed_notification и удаляет чат»
    - Что: при окончательном провале отправки → пометить чат как неактивный, уведомить админа
    - Почему: сейчас чат остаётся в списке, но сообщения туда не доходят — молчаливая потеря

#### P3 — Расширение функциональности (дальняя перспектива)

11. **Web-админка (FastAPI /admin)**
    - Источник: FULL_TZ точки расширения
    - Что: REST API для управления курсами, заявками, пользователями + простая веб-страница
    - Почему: удобнее, чем через Telegram-меню

12. **Мониторинг (Prometheus + Grafana)**
    - Источник: risks.md п.5
    - Что: метрики (количество заявок, время ответа, ошибки), Grafana дашборд
    - Почему: операционная видимость

13. **Интеграция с криптобиржей**
    - Источник: FULL_TZ точки расширения
    - Что: ExchangeService — автоматическая проверка поступления, курс из API
    - Почему: устранение ручного подтверждения оператором

14. **Мультивалютность**
    - Источник: FULL_TZ точки расширения
    - Что: расширение order_type, динамические реквизиты по валюте
    - Почему: масштабирование бизнеса

15. **Другие каналы уведомлений**
    - Источник: FULL_TZ точки расширения
    - Что: email, SMS через NotificationService
    - Почему: redundancy каналов связи

## Реализация

Создать файл `docs/roadmap.md` с содержанием, описанным выше. Формат:
- Заголовок + вводный абзац (текущий статус: MVP завершён, бот работает)
- Таблица всех пунктов с приоритетом, источником, статусом
- Детальное описание каждого пункта (что, почему, пример реализации)
- Отдельно: «Пост-деплойные фиксы» — баги, найденные при первом деплое (enum name=, ARQ RedisSettings, ARQ_REDIS_URL)
