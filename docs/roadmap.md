# Roadmap пост-MVP развития

## Текущий статус

MVP завершён — все 7 фаз реализованы (см. `status.md`). Бот работает в Docker Compose: Long Polling, PostgreSQL, Redis, ARQ Worker. Реализовано 16 handlers, 6 сервисов, шифрование реквизитов, уведомления через ARQ.

Данный roadmap описывает задачи, необходимые для перевода проекта из MVP в production-ready состояние, а также направления дальнейшего развития.

## P0 — Критично для production

Задачи, без которых нельзя деплоить в production. Каждая решает реальный риск потери данных или недоступности сервиса.

### 1. Graceful shutdown (SIGTERM)

**Источник:** `risks.md` п.3, анализ `app/main.py`

**Проблема:** `main.py` содержит только `try/finally` с закрытием bot session. При Docker `restart` или `stop` процесс получает SIGTERM, но не обрабатывает его — транзакции БД могут оборваться посередине, Long Polling прерывается без чистого завершения.

**Что сделать:**
- Зарегистрировать обработчик SIGTERM в `app/main.py`
- При сигнале: остановить `dp.start_polling()`, закрыть `bot.session`, вызвать `await async_engine.dispose()`
- См. пример в `risks.md` п.3

**Почему критично:** Без этого Docker `restart` убивает процесс посреди транзакции → возможны частичные записи, зависшие подключения к БД.

---

### 2. Глобальный обработчик ошибок aiogram

**Источник:** `FULL_TZ.md` Этап 10, анализ `app/bot.py`

**Проблема:** В `app/bot.py` на Dispatcher не зарегистрирован `@dp.errors` (или аналог). Необработанные исключения в handlers молча проглатываются aiogram — пользователь не получает ответа, ошибка не логируется с контекстом.

**Что сделать:**
- Зарегистрировать error handler на Dispatcher: `dp.errors.register(error_handler)`
- В обработчике: логировать исключение с `update`, `exception`, отправлять пользователю «Произошла ошибка, попробуйте позже»
- Различать пользовательские ошибки (FSM валидация — не логировать как error) от системных (DB, сеть — логировать как error)

**Почему критично:** Без этого любая непредвиденная ошибка в handler'е приводит к «молчаливому зависанию» — пользователь ждёт ответа, но не получает его.

---

### 3. Проверка `is_blocked` в UserMiddleware

**Источник:** Модель `app/database/models/user.py:25` — поле `is_blocked` существует, анализ `app/middlewares/user_middleware.py`

**Проблема:** Поле `is_blocked` объявлено в модели User и существует в миграции, но нигде не проверяется. `UserMiddleware` загружает пользователя в `data["user"]`, но не проверяет флаг блокировки. Ни один handler, ни один middleware не реагирует на `is_blocked=True`.

**Что сделать:**
- В `UserMiddleware.__call__` после загрузки пользователя: если `user.is_blocked is True` — отправить «Вы заблокированы. Обратитесь в поддержку.» и не вызывать `handler`
- Для `CallbackQuery` — ответить через `event.answer()`, для `Message` — через `event.answer()`
- Admin/SuperAdmin не блокируются (даже если флаг стоит)

**Почему критично:** Без этого нет механизма блокировки злоупотребляющих клиентов — единственный способ удалить пользователя из БД, что нарушает целостность (FK в orders).

## P1 — Важно для стабильности

Задачи, снижающие риск инцидентов и улучшающие устойчивость к ошибкам.

### 4. Обработка потери соединения с БД

**Источник:** `FULL_TZ.md` секция 7

**Проблема:** Если PostgreSQL недоступен, `DBSessionMiddleware` выбросит исключение при создании сессии — необработанное, оно дойдёт до aiogram и будет молча проглочено (см. п.2). Пользователь не получит никакого ответа.

**Что сделать:**
- В `DBSessionMiddleware.__call__` обернуть создание сессии в `try/except`
- При ошибке подключения: логировать error, отправить пользователю «Сервис временно недоступен, попробуйте позже», не вызывать `handler`
- Альтернатива: полагаться на глобальный error handler (п.2), но тогда пользователю всё равно нужен ответ

**Ссылка:** `docs/scenarios.md` — текущие сценарии не описывают поведение при падении БД.

---

### 5. ThrottlingMiddleware (антиспам)

**Источник:** `FULL_TZ.md` Этап 4 п.3

**Проблема:** Сейчас любой пользователь может отправлять сообщения с любой частотой. Нет защиты от спама — как злонамеренного, так и случайного (зажатая кнопка, автоповтор).

**Что сделать:**
- Создать `app/middlewares/throttling.py` — `ThrottlingMiddleware`
- Лимиты: 1 сообщение/сек для команд (`/start` и т.д.), 5 сообщений/мин для FSM-ввода
- Механизм: in-memory dict с `time.monotonic()` (аналогично `_bot_enabled_cache` в `bot_status.py`), или Redis для многозадачности
- При превышении: молча игнорировать или отправить «Слишком много запросов, подождите»

**Ссылка:** `docs/scenarios.md` — все 16 handlers не защищены от спама.

---

### 6. Лимит попыток FSM-ввода (3 попытки)

**Источник:** `FULL_TZ.md` секция 7: «Максимум 3 попытки, затем выход»

**Проблема:** При невалидном вводе в FSM (сумма, дата, Telegram ID) пользователь может бесконечно вводить неверные данные. FSM не сбрасывается автоматически — он остаётся в состоянии `waiting_amount` / `waiting_new_rate` и т.д., пока пользователь не введёт корректное значение или не нажмёт `/start`.

**Что сделать:**
- В каждом FSM handler'е с валидацией: хранить счётчик попыток в `data` (или `state.proxy`)
- После 3 невалидных вводов: сбросить FSM (`await state.clear()`), отправить «Слишком много ошибок. Начните заново.»
- Затронутые FSM: `OrderBuyStates`, `OrderSellStates`, `ChangeRateStates`, `ChangeLinksStates`, `StatisticsStates`, `AssignRoleStates`, `SupportStates` (см. `app/fsm/`)

**Ссылка:** `docs/scenarios.md` — все FSM-сценарии не имеют ограничения попыток.

## P2 — Полезно для операционной эффективности

Задачи, улучшающие наблюдаемость, масштабируемость и UX, но не блокирующие production.

### 7. Redis-кеш флагов вместо in-memory dict

**Источник:** `risks.md` п.1, `architecture.md`, анализ `app/middlewares/bot_status.py`

**Проблема:** `BotStatusMiddleware` (`bot_status.py:16`) использует in-memory dict `_bot_enabled_cache` с TTL 30 сек. Документация `architecture.md` описывает Redis-кеш флагов, но реализация — in-memory. При запуске нескольких инстансов бота (горизонтальное масштабирование) кеш не синхронизируется между процессами.

**Что сделать:**
- Заменить `_bot_enabled_cache` dict на чтение из Redis (база 0, ключ `bot_enabled` / `buy_enabled` / `sell_enabled`, TTL 30 сек)
- При записи флага (админ toggles) — писать в Redis напрямую + PostgreSQL
- В middleware — читать из Redis, при miss — читать из PostgreSQL и кешировать в Redis
- Использовать существующее подключение Redis (см. `config.py`: `REDIS_URL`)

**Риск:** При потере Redis — fallback на PostgreSQL (как сейчас).

---

### 8. Структурированное JSON-логирование

**Источник:** `risks.md` п.2

**Проблема:** `app/main.py` использует `logging.basicConfig` с текстовым форматом. В Docker-окружении логи неудобно парсить и агрегировать (Grafana Loki, ELK).

**Что сделать:**
- Заменить `logging.basicConfig` на JSON-форматтер (см. пример в `risks.md` п.2)
- Поля: `timestamp`, `level`, `message`, `module`, `user_id` (если есть в контексте)
- Не добавлять новые зависимости — реализовать через `logging.Formatter`

---

### 9. Healthcheck HTTP-эндпоинт для bot-контейнера

**Источник:** `risks.md` п.4

**Проблема:** Docker не может определить, жив ли бот — сейчас только `restart: unless-stopped`. При зависании Long Polling (network issue) контейнер продолжает работать, но не обрабатывает сообщения.

**Что сделать:**
- Запустить aiohttp web на порту 8080 рядом с Long Polling (в том же процессе)
- `GET /health` → 200 OK + проверка DB ping (`SELECT 1`)
- Добавить `healthcheck` в `docker-compose.yml` для сервиса `bot` (см. пример в `risks.md` п.4)

**Ссылка:** `architecture.md` — Bot компонент не имеет healthcheck.

---

### 10. Улучшение обработки уведомлений (failed_notification)

**Источник:** `FULL_TZ.md` секция 7: «При исчерпании попыток логгирует `failed_notification` и удаляет чат»

**Проблема:** При окончательном провале отправки (ARQ max_tries=3 исчерпан) чат остаётся в таблице `notification_chats`. Следующие уведомления будут снова пытаться отправить в недоступный чат — молчаливая потеря событий. Админ не уведомляется о проблеме.

**Что сделать:**
- В ARQ job `send_notification`: при `max_tries` исчерпан — пометить `notification_chat.is_active = False` (потребует добавить поле в модель и миграцию)
- Уведомить админа (в другой чат или напрямую SuperAdmin): «Чат {chat_id} недоступен, уведомления приостановлены»
- В `NotificationService` — выбирать только чаты с `is_active = True`

## P3 — Расширение функциональности (дальняя перспектива)

Фичи из точек расширения `FULL_TZ.md`. Не требуются для текущей работы бота, но важны для масштабирования бизнеса.

### 11. Web-админка (REST API)

**Источник:** `FULL_TZ.md` точки расширения

**Описание:** REST API для управления курсами, заявками, пользователями + простая веб-страница. Удобнее, чем Telegram-меню, для массовых операций и аналитики. В текущей архитектуре — отдельный сервис или модуль внутри bot-процесса.

**Сложность:** Высокая. Требует авторизации, отдельного HTTP-сервера, CSRF-защиты.

---

### 12. Мониторинг (Prometheus + Grafana)

**Источник:** `risks.md` п.5

**Описание:** Метрики (количество заявок, время ответа, ошибки API), Grafana дашборд, алерты при падении сервисов. Даёт операционную видимость — без мониторинга инциденты обнаруживаются только по жалобам пользователей.

**Сложность:** Средняя. Требует добавления Prometheus-метрик в код и настройки Grafana.

---

### 13. Интеграция с криптобиржей

**Источник:** `FULL_TZ.md` точки расширения

**Описание:** `ExchangeService` — автоматическая проверка поступления платежей, курс из API биржи. Устраняет ручное подтверждение оператором.

**Сложность:** Высокая. Требует интеграции с API биржи, обработки транзакций, безопасности ключей.

---

### 14. Мультивалютность

**Источник:** `FULL_TZ.md` точки расширения

**Описание:** Расширение `order_type` (добавление BTC, ETH и др.), динамические реквизиты по валюте, отдельные курсы.

**Сложность:** Высокая. Затрагивает модели, FSM, клавиатуры, уведомления — практически все слои приложения.

---

### 15. Другие каналы уведомлений (email, SMS)

**Источник:** `FULL_TZ.md` точки расширения

**Описание:** Расширение `NotificationService` для отправки уведомлений через email, SMS. Redundancy каналов связи — если Telegram недоступен, админ всё равно получит уведомление.

**Сложность:** Средняя. Требует интеграции с SMTP / SMS-провайдером.

## Пост-деплойные фиксы

Баги, обнаруженные и устранённые при первом деплое. Задокументированы для предотвращения повторения.

### Исправлено

| # | Баг | Причина | Исправление | Файл |
|---|-----|---------|-------------|------|
| 1 | Enum `name=` mismatch | ORM `Enum(RoleEnum)` генерирует имя типа `roleenum`, но миграция создавала `user_role` | Добавлен явный `name="user_role"`, `name="order_type"`, `name="order_status"`, `name="rate_type"` во все `Enum()` | `app/database/models/*.py` |
| 2 | ARQ `RedisSettings` перемещён | В ARQ v0.26.x `RedisSettings` перенесён из `arq` в `arq.connections` | Импорт изменён на `from arq.connections import RedisSettings` | `app/tasks/worker.py` |
| 3 | `ARQ_REDIS_URL` не работает в Docker | Значение по умолчанию `redis://localhost:6379/1` не резолвится внутри контейнера | В `docker-compose.yml` добавлено `ARQ_REDIS_URL=redis://redis:6379/1` в environment | `docker-compose.yml` |

### Уроки

1. **Enum name=** — всегда указывать `name=` явно при объявлении `Enum()` в SQLAlchemy, чтобы избежать расхождения между ORM и миграцией.
2. **Docker DNS** — в `docker-compose.yml` сервисы обращаются друг к другу по именам сервисов (`redis`, `postgres`), не по `localhost`. Все URL подключения в `.env` / environment должны использовать имена сервисов.
3. **ARQ API-дрифт** — при обновлении ARQ проверять changelog на предмет перемещения импортов.

## Порядок реализации

```
P0 (блокирующие)     → P1 (стабильность)    → P2 (эффективность)  → P3 (расширение)
─────────────────────────────────────────────────────────────────────────────────
1. Graceful shutdown    4. DB error handling    7. Redis-кеш          11. Web-админка
2. Error handler        5. Throttling           8. JSON-логи          12. Мониторинг
3. is_blocked check     6. FSM attempt limits   9. Healthcheck       13. Криптобиржа
                                                                       14. Мультивалютность
                                               10. Notifications     15. Email/SMS
```

P0 рекомендуется реализовать в первую очередь — без них production-деплой несёт операционные риски. P1 — следующий приоритет для устойчивости. P2 и P3 — по мере необходимости.
