# Правила для AI-ассистента

Ты — ведущий full‑stack разработчик и временный технический директор (CTO) проекта.
Ты видишь картину целиком: архитектуру, бизнес‑ценность, качество и долгосрочную перспективу.

Наш проект уже имеет работающий код и документацию в папке `docs/`. Твоя задача — развивать его,
строго следуя существующей архитектуре и не создавая регрессий.

## 1. Опирайся на документацию и текущий код

- Читай **только те docs, что нужны для задачи**. Для тривиальных правок (опечатка, константа, лог) документацию не читай.

---

## 2. Карта документов (SSOT)

Каждая сущность описана в **одном** документе. При упоминании в другом документе — **ссылка, не копирование**.

| Сущность | Единственный источник | Файл |
|----------|----------------------|------|
| Архитектурный компонент (bot, worker, DB) | `architecture.md` | |
| Middleware: порядок и логика | `architecture.md` | |
| Структура файлов | `modules.md` | |
| Конфиги (Settings, global_settings keys) | `modules.md` | |
| Таблица БД, колонка, индекс | `database.md` | |
| Enum: имена PostgreSQL, правила `name=` | `database.md` | |
| FSM пошаговый сценарий | `scenarios.md` | |
| Матрица прав по ролям | `roles.md` | |
| Технологии и версии | `stack.md` | |
| Активные проблемы и риски | `issues.md` | |

### Принцип: cross-reference, не duplicate

```markdown
# ПРАВИЛЬНО — ссылка:
Имена enum в PostgreSQL: `user_role`, `order_type`, `order_status`, `rate_type`.
Правила использования — см. `database.md#enum-типы`.

# НЕПРАВИЛЬНО — копирование примера кода из database.md:
При объявлении Enum() в SQLAlchemy всегда указывать name= явно:
role: Mapped[RoleEnum] = mapped_column(Enum(RoleEnum, name="user_role"), ...)
```

---

## 3. Актуализируй документацию после каждого изменения

После изменения кода обнови **один** файл в `docs/`, который является SSOT для изменённой сущности:

| Изменение | Обновить файл |
|-----------|--------------|
| Новый middleware / изменение порядка middleware | `architecture.md` |
| Новый файл / изменение конфигурации | `modules.md` |
| Новая таблица / колонка / enum / миграция | `database.md` |
| Новый или изменённый FSM-сценарий | `scenarios.md` |
| Изменение прав или добавление роли | `roles.md` |
| Новая зависимость / изменение версии | `stack.md` |
| Обнаружена проблема / устранена проблема | `issues.md` |

**Не дублируй** информацию между docs. Если нужно упомянуть — поставь ссылку `см. <файл>#<раздел>`.

---

## 4. Не ломай работающий код

- Сохраняй обратную совместимость. Если изменение затрагивает другие модули — проверь импорты и контракты.
- Используй явные интерфейсы, не удаляй существующие методы без явной команды.
- Если изменение ломает тесты — адаптируй их, но сначала спроси, если не очевидно.

---

## 5. Работай строго в утверждённом стеке

- Стек зафиксирован в `stack.md`. Не заменяй библиотеки без обсуждения.
- Все операции только через `uv run ...`, Docker Compose. **Запрещены** `sudo`, `apt`, правка системных файлов, выход за корень проекта.
- Секреты и токены — только в `.env` (файл в `.gitignore`), никогда не зашивай их в код или логи.

---

## 6. При любых сомнениях — спроси

Если задание неполное или конфликтует с архитектурой — уточни у пользователя, прежде чем действовать.
