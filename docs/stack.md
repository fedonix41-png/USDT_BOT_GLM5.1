# Технологический стек

## Основные технологии

| Технология | Версия | Роль в проекте |
|-----------|--------|----------------|
| Python | 3.11+ | Основной язык программирования. Выбран за поддержку async/await, type hints и богатую экосистему. |
| Aiogram | 3.15.x | Фреймворк для Telegram Bot API. Обеспечивает Long Polling, FSM (Finite State Machine), middleware, роутинг, inline-клавиатуры, callback-обработку. Основной каркас бота. |
| SQLAlchemy | 2.0.x | ORM и система запросов к БД. Используется в асинхронном режиме (`sqlalchemy[asyncio]`) с typed mapping (`Mapped[]`). Определение моделей, запросы через `select()`, `insert()`, агрегации. |
| asyncpg | 0.30.x | Асинхронный драйвер PostgreSQL для Python. Работает в связке с SQLAlchemy async engine для неблокирующего доступа к БД. |
| PostgreSQL | 15-alpine | Реляционная СУБД. Хранит все данные: пользователи, заявки, курсы, настройки, чаты уведомлений, аудит-логи. Выбрана за надёжность, поддержку JSONB, индексов и транзакций. Образ `alpine` — минимальный размер. |
| Redis | 7-alpine | Хранилище ключ-значение в памяти. Две функции: (1) кеш флагов `bot_enabled`, `buy_enabled`, `sell_enabled` с TTL 30 сек; (2) очередь задач для ARQ (база данных 1). |
| ARQ | 0.26.x | Асинхронная очередь задач на базе Redis. Фоновый воркер обрабатывает задачи: отправка уведомлений, обновление битых ссылок. Поддерживает retry, max_tries, job timeout. |
| Pydantic Settings | 2.7.x | Управление конфигурацией приложения. Чтение переменных окружения из `.env`, валидация типов, значения по умолчанию. Класс `Settings` содержит `BOT_TOKEN`, `DATABASE_URL`, `REDIS_URL`, `ENCRYPTION_KEY`, `SUPER_ADMIN_TELEGRAM_ID`. |
| Cryptography | 44.0.x | Библиотека шифрования. Используется для AES-256-CBC шифрования реквизитов оплаты (`payment_link_buy`, `payment_link_sell`). IV генерируется случайно, prepend к ciphertext, результат — hex-строка. |
| Alembic | 1.14.x | Инструмент миграций БД. Управление схемой PostgreSQL: создание таблиц, индексов, enum-типов. Команда `alembic upgrade head` применяет все миграции. Первая миграция создаёт 6 таблиц. |
| Docker | — | Контейнеризация. Каждый компонент (bot, arq-worker, postgres, redis) работает в изолированном контейнере. Обеспечивает воспроизводимость окружения. |
| Docker Compose | 3.9 | Оркестрация контейнеров. Определяет сервисы, их зависимости, healthcheck, volumes, переменные окружения. Управляет порядком запуска. |
| uv | latest | Быстрый менеджер пакетов Python (замена pip/poetry). Устанавливает зависимости из `pyproject.toml` и `uv.lock`. Используется внутри Docker-контейнера и для локальной разработки. |

## Выбор технологий: обоснование

### Long Polling вместо Webhook

Webhook требует публичного HTTPS-эндпоинта и дополнительного веб-сервера (FastAPI). Long Polling проще в развёртывании на одном VPS, не требует открытия портов, достаточно надёжен для MVP. При перезапуске бота есть окно ~30 сек возможной потери обновлений — допустимо для проекта.

### AES-256-CBC для реквизитов

Реквизиты оплаты — чувствительные данные. Шифрование на уровне приложения (не БД) позволяет хранить в БД зашифрованный текст. Ключ шифрования — в `.env`, не в коде. При компрометации БД реквизиты остаются защищёнными.

### ARQ вместо Celery

ARQ легче, работает на Redis (уже используется для кеша), полностью асинхронный (не блокирует event loop). Celery избыточен для 2 фоновых задач и требует дополнительного брокера (RabbitMQ).

### SQLAlchemy 2.0 с typed mapping

`Mapped[]` и `mapped_column()` обеспечивают типобезопасность моделей и автодополнение в IDE. Асинхронный режим (`asyncpg`) не блокирует event loop при запросах к БД.

### Pydantic Settings

Единая точка конфигурации: все переменные окружения валидируются при старте приложения. Ошибка в `.env` — немедленный сбой с понятным сообщением, а не загадочная ошибка в runtime.
