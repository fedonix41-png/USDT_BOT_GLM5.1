# Риски и рекомендации

## Активные риски

### 1. AES-256-CBC и ключ шифрования
**Риск:** При потере `ENCRYPTION_KEY` зашифрованные реквизиты невозможно восстановить.

**Меры:**
- Резервное копирование `.env` в безопасное место
- Документирование процедуры восстановления

---

### 2. Telegram API rate limits
**Риск:** При массовой рассылке возможен ответ 429 (Too Many Requests).

**Реализовано:**
- ARQ retry с exponential backoff (max_tries=3)
- Логирование ошибок

---

### 3. Отсутствие курса при первом запуске
**Риск:** Если админ не задал курс, клиент увидит «Курс не установлен».

**Реализовано:**
- Корректная обработка `None` в RateService
- Понятное сообщение для пользователя

---

## Требующие реализации

### 4. Long Polling vs Webhook
**Риск:** При перезапуске бота есть окно ~30 секунд, когда обновления могут быть пропущены.

**Решение:** Для MVP допустимо. В будущем — переход на Webhook.

### 5. Graceful shutdown
**Риск:** При Docker restart транзакции БД могут оборваться.

**Решение:** Обработчик SIGTERM в `main.py` (см. roadmap.md п.1).

### 6. Глобальный error handler
**Риск:** Необработанные исключения молча проглатываются.

**Решение:** `@dp.errors()` в `bot.py` (см. roadmap.md п.2).

---

## Рекомендации (не реализовано)

### Рекомендация для Подмена номера телефона при создании заявки: при желании можно усилить защиту, добавив учёт попыток подмены через check_fsm_attempts() (как в других FSM-обработчиках), чтобы после 3 неудачных попыток сценарий автоматически сбрасывался. На текущий момент это необязательно, так как простой return позволяет пользователю сразу повторить попытку с правильным контактом.

### Redis-кеш для флагов
Заменить in-memory dict в `bot_status.py` на Redis. Снижение нагрузки на PostgreSQL, синхронизация при масштабировании.

### Структурированное логирование
JSON-формат логов для Docker (Grafana Loki, ELK).

### Healthcheck для bot-контейнера
HTTP-эндпоинт `/health` для определения живости бота.

### Мониторинг (на будущее)
- Prometheus метрики
- Grafana дашборд
- Алерты при падении сервисов

## Устранённые риски (Аудит Май 2026)

### 1. Полный обход авторизации (Broken Access Control / IDOR)
**Было:** Обработчики операторов и администраторов не проверяли `user.role`, что позволяло клиентам вызывать функции через подделку Callback Data.
**Решение:** Внедрена жесткая проверка `user.role` во все административные и операторские хэндлеры (`active_orders`, `complete_order`, `management`, и т.д.).

### 2. Некорректная обработка состояния FSM (Finite State Machine)
**Было:** Отсутствовала проверка роли внутри обработчиков изменения состояния (например, при сохранении курса валют).
**Решение:** В каждый обработчик `FSM` добавлена проверка роли на соответствие уровню `admin` или `super_admin`.

### 3. IDOR в жалобе на битую ссылку (`broken_link.py`)
**Было:** Любой пользователь мог отправить callback `order_broken_link:<ID>` и пометить ссылку в чужом заказе как нерабочую.
**Решение:** Добавлена проверка `order.user_id == user.id` — только владелец заказа может сообщить о проблеме со ссылкой.

### 4. Отсутствие логирования попыток атак
**Было:** При попытке несанкционированного доступа (подделка callback data) события не логировались, что затрудняло мониторинг атак.
**Решение:** Во все защищённые обработчики добавлено логирование через `logger.warning()` с указанием:
- `user_id` атакующего
- `callback` / `command` — вектор атаки
- `required_role` — требуемая роль для доступа

### 5. Подмена номера телефона при создании заявки (P0 — критично)
**Было:** Пользователь мог переслать контакт другого человека вместо своего. Telegram позволяет пересылать контакты третьих лиц, и код не проверял принадлежность контакта.
**Файлы:** `app/handlers/client/buy.py`, `app/handlers/client/sell.py`
**Решение:** Добавлена проверка `message.contact.user_id != message.from_user.id` в обработчики `process_buy_phone` и `process_sell_phone`. При несовпадении — отказ с сообщением "Пожалуйста, поделитесь своим контактом." и логирование через `logger.warning()`.

## Обнаруженные проблемы (Аудит 27.05.2026)

### 1. Self-ban защита отсутствует (P1)
**Проблема:** Админ может случайно забанить сам себя, введя свой Telegram ID.

**Файл:** `app/handlers/admin/ban_user.py:41-45`

**Решение:** Добавить проверку:
```python
if target_telegram_id == message.from_user.id:
    await message.answer("Невозможно заблокировать самого себя.")
    await state.clear()
    return
```

---

### 2. Отсутствие кнопки отмены на шаге запроса телефона (P1 — UX)
**Проблема:** Пользователь без username при создании заявки видит только кнопку «📱 Поделиться номером». Нет возможности отменить операцию, хотя глобальный cancel handler зарегистрирован.

**Файлы:** `app/handlers/client/buy.py:143-151`, `app/handlers/client/sell.py:142-150`

**Решение:** Добавить кнопку отмены в ReplyKeyboardMarkup:
```python
phone_kb = ReplyKeyboardMarkup(
    keyboard=[
        [KeyboardButton(text="📱 Поделиться номером", request_contact=True)],
        [KeyboardButton(text="❌ Отмена")],
    ],
    resize_keyboard=True,
)
```

---

### 3. Inconsistency в использовании check_fsm_attempts() (P2)
**Проблема:** Разные обработчики используют `check_fsm_attempts()` по-разному:
- `ban_user.py` игнорирует возвращаемое значение
- `assign_roles.py` использует `should_continue`

**Файлы:** 
- `app/handlers/admin/ban_user.py:36-39` — не использует возвращаемое значение
- `app/handlers/admin/assign_roles.py:36-39` — использует `should_continue`

**Решение:** Унифицировать использование во всех FSM-обработчиках:
```python
should_continue, _ = await check_fsm_attempts(state, message, "...")
return
```

---

### 4. reset_fsm_attempts() не используется (P2)
**Проблема:** Функция `reset_fsm_attempts()` определена в `helpers.py`, но нигде не вызывается. Это может привести к накоплению счётчика попыток между разными FSM-сценариями.

**Файл:** `app/utils/helpers.py:51-56`

**Решение:** Вызывать `reset_fsm_attempts()` перед началом каждого FSM-сценария:
```python
await reset_fsm_attempts(state)
await state.set_state(SomeStates.waiting_input)
```
