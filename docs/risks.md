# Риски и рекомендации

## Активные риски

### 1. AES-256-CBC и ключ шифрования
**Риск:** При потере `ENCRYPTION_KEY` зашифрованные реквизиты невозможно восстановить.

**Меры:**
- Резервное копирование `.env` в безопасное место
- Документирование процедуры восстановления

---

### 2. Telegram API rate limits
**Риск:** При массовой рассылке возможен ответ 429 (Too Many Requests).

**Реализовано:**
- ARQ retry с exponential backoff (max_tries=3)
- Логирование ошибок

---

### 3. Отсутствие курса при первом запуске
**Риск:** Если админ не задал курс, клиент увидит «Курс не установлен».

**Реализовано:**
- Корректная обработка `None` в RateService
- Понятное сообщение для пользователя

---

## Требующие реализации

### 4. Long Polling vs Webhook
**Риск:** При перезапуске бота есть окно ~30 секунд, когда обновления могут быть пропущены.

**Решение:** Для MVP допустимо. В будущем — переход на Webhook.

### 5. Graceful shutdown
**Риск:** При Docker restart транзакции БД могут оборваться.

**Решение:** Обработчик SIGTERM в `main.py` (см. roadmap.md п.1).

### 6. Глобальный error handler
**Риск:** Необработанные исключения молча проглатываются.

**Решение:** `@dp.errors()` в `bot.py` (см. roadmap.md п.2).

---

JSON-формат логов для Docker (Grafana Loki, ELK).

### Healthcheck для bot-контейнера

## Устранённые риски (Аудит Май 2026)

### 1. Полный обход авторизации (Broken Access Control / IDOR)
**Было:** Обработчики операторов и администраторов не проверяли `user.role`, что позволяло клиентам вызывать функции через подделку Callback Data.
**Решение:** Внедрена жесткая проверка `user.role` во все административные и операторские хэндлеры (`active_orders`, `complete_order`, `management`, и т.д.).

### 2. Некорректная обработка состояния FSM (Finite State Machine)
**Было:** Отсутствовала проверка роли внутри обработчиков изменения состояния (например, при сохранении курса валют).
**Решение:** В каждый обработчик `FSM` добавлена проверка роли на соответствие уровню `admin` или `super_admin`.

### 3. IDOR в жалобе на битую ссылку (`broken_link.py`)
**Было:** Любой пользователь мог отправить callback `order_broken_link:<ID>` и пометить ссылку в чужом заказе как нерабочую.
**Решение:** Добавлена проверка `order.user_id == user.id` — только владелец заказа может сообщить о проблеме со ссылкой.

### 4. Отсутствие логирования попыток атак
**Было:** При попытке несанкционированного доступа (подделка callback data) события не логировались, что затрудняло мониторинг атак.
**Решение:** Во все защищённые обработчики добавлено логирование через `logger.warning()` с указанием:
- `user_id` атакующего
- `callback` / `command` — вектор атаки
- `required_role` — требуемая роль для доступа
