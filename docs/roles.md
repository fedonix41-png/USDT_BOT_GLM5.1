# Ролевая модель

## Иерархия ролей

Система реализует строгую иерархию с 4 уровнями доступа:

```
super_admin  (уровень 4) — максимальные права
    │
    ├── admin  (уровень 3)
    │       │
    │       ├── operator  (уровень 2)
    │       │       │
    │       │       └── client  (уровень 1) — базовая роль
```

**Правило наследования:** каждая роль включает все права нижестоящих ролей, плюс собственные уникальные права.

## Меню по ролям

### Client (Клиент)

Базовая роль. Все новые пользователи получают её при `/start`.

```
[💰 Купить] [💸 Продать]
[📊 Курсы] [📞 Поддержка]
```

**Доступные действия:**
- Покупка USDT (FSM — ввод суммы, подтверждение)
- Продажа USDT (FSM — ввод суммы, подтверждение)
- Просмотр текущих курсов покупки и продажи
- Отмена своей заявки (inline-кнопка под сообщением с заявкой)
- Жалоба на неработающую ссылку (inline-кнопка)
- Обращение в поддержку (FSM — ввод текста)

**Ограничения:**
- Если `buy_enabled=False` → кнопка «Купить» заменяется на «🛑 Закуп остановлен»
- Если `sell_enabled=False` → кнопка «Продать» заменяется на «🛑 Продажа остановлена»
- Если `bot_enabled=False` → все клиентские действия блокируются с сообщением «Бот временно недоступен»

---

### Operator (Оператор)

Роль для обработки клиентских заявок. Не может создавать заявки от своего имени.

```
[📋 Заявки] [📊 Курсы]
[📈 Статистика]
```

**Доступные действия:**
- Просмотр всех активных заявок (статус `created`, последние 24 часа) с пагинацией
- Завершение заявки (inline-кнопка «✅ Завершить») — клиент получает уведомление
- Отмена заявки (inline-кнопка «❌ Отменить»)
- Просмотр статистики за период (FSM — ввод дат)
- Просмотр текущих курсов

**Особенности:**
- Оператор видит заявки всех клиентов, а не только свои
- При завершении заявки клиенту отправляется чек-уведомление
- Операторы работают даже когда бот отключён для клиентов (`bot_enabled=False`)

---

### Admin (Администратор)

Полный управленческий доступ, кроме назначения других админов.

**Reply-клавиатура:**

```
[📋 Заявки] [📈 Статистика] [📊 Курсы]
[⚙️ Управление]
```

**Inline-панель управления** (открывается при нажатии ⚙️ Управление):

```
[🔄 Курс покупки] [🔄 Курс продажи]
[🔗 Реквизиты]
[✅ Закуп вкл / ⏸ Стоп закуп] [✅ Продажа вкл / ⏸ Стоп продажа]
[🛑 Отключить бота / ✅ Включить бота]
[➕ Чаты] [👤 Оператор]
[🔙 Закрыть]
```

**Доступные действия (помимо прав Operator):**
- Изменение курса покупки (FSM — запуск из inline-кнопки `🔄 Курс покупки`, ввод нового значения)
- Изменение курса продажи (FSM — запуск из inline-кнопки `🔄 Курс продажи`, ввод нового значения)
- Смена реквизитов для покупки/продажи (FSM — запуск из inline-кнопки `🔗 Реквизиты`, выбор типа, ввод ссылки)
- Стоп/старт закупа (inline-кнопка-переключатель, инверсия флага `buy_enabled`)
- Стоп/старт продажи (inline-кнопка-переключатель, инверсия флага `sell_enabled`)
- Глобальное отключение/включение бота (inline-кнопка-переключатель, инверсия `bot_enabled`)
- Управление чатами уведомлений (добавление, удаление, просмотр списка)
- Назначение роли «Оператор» другим пользователям (FSM — запуск из inline-кнопки `👤 Оператор`, ввод Telegram ID)

**Особенности:**
- Все действия админа логируются в `audit_logs`
- При смене реквизитов — автоматическое обновление сообщений клиентов с битой ссылкой (через ARQ)
- Кнопки-переключатели обновляются inline (без перерисовки всей клавиатуры)
- FSM-действия (смена курса, реквизитов, назначение роли) запускаются из inline-колбэков панели управления

---

### SuperAdmin (Суперадминистратор)

Максимальные права. Единственный, кто может назначать админов.

**Reply-клавиатура:**

```
[📋 Заявки] [📈 Статистика] [📊 Курсы]
[⚙️ Управление]
```

**Inline-панель управления** (открывается при нажатии ⚙️ Управление):

```
[🔄 Курс покупки] [🔄 Курс продажи]
[🔗 Реквизиты]
[✅ Закуп вкл / ⏸ Стоп закуп] [✅ Продажа вкл / ⏸ Стоп продажа]
[🛑 Отключить бота / ✅ Включить бота]
[➕ Чаты] [👤 Оператор]
[👑 Админ]
[🔙 Закрыть]
```

**Дополнительные права (помимо Admin):**
- Назначение роли «Админ» другим пользователям (FSM — запуск из inline-кнопки `👑 Админ`, ввод Telegram ID)

**Особенности:**
- SuperAdmin назначается при первом запуске через `SUPER_ADMIN_TELEGRAM_ID` в `.env`
- Это единственная роль, которую невозможно назначить через интерфейс бота
- Кнопка `👑 Админ` видна только SuperAdmin в inline-панели управления

## Процессы назначения ролей

### При регистрации (/start)

```
Пользователь → /start
│
├─ Проверить: пользователь в БД?
│  ├─ Нет → создать с role='client'
│  │        (если telegram_id == SUPER_ADMIN_TELEGRAM_ID → role='super_admin')
│  └─ Да → загрузить существующую роль
│
└─ Отправить меню по роли
```

**Критически важно:** пользователь должен сначала запустить бота через `/start`, прежде чем ему можно назначить другую роль. Иначе его нет в БД и назначение невозможно.

### Назначение Оператора (Admin или SuperAdmin)

```
Admin/SuperAdmin → [⚙️ Управление] → [👤 Оператор]
│
├─ Callback: mgmt:assign_operator
│
├─ FSM: AssignRoleStates.waiting_target_user
│  «Введите Telegram ID пользователя или перешлите его контакт:»
│
├─ Пользователь вводит Telegram ID
│  ├─ ID найден в БД → установить role='operator'
│  └─ ID не найден → «Пользователь не найден. Он должен сначала запустить бота (/start).»
│
├─ Логирование: audit_logs (action='assign_role', details={'role': 'operator'})
│
├─ Уведомление пользователю: «👤 Вы назначены Оператором бота обмена USDT.»
│
├─ Уведомление админу: «✅ Пользователь @{username} теперь Оператор.»
│
└─ Уведомление в чаты через ARQ
```

### Назначение Админа (только SuperAdmin)

Аналогично назначению оператора, но:
- Кнопка `👑 Админ` доступна только SuperAdmin (в inline-панели управления)
- Callback: `mgmt:assign_admin`
- Устанавливается `role='admin'`
- Проверка через `RoleGuardMiddleware`: `min_role=RoleEnum.super_admin`
- При попытке Admin'а назначить другого админа → «У вас нет прав для этого действия»

### Проверка прав в Handlers и Middleware

В дополнение к базовой защите через `RoleGuardMiddleware`, в проекте внедрены **жёсткие проверки ролей на уровне каждого обработчика** (message и callback_query). Это устраняет уязвимости IDOR (Broken Access Control) и предотвращает вызовы административных действий через подделку Callback Data или прямую манипуляцию FSM-стейтами.

Пример проверки в хэндлере:
```python
@router.callback_query(F.data == "mgmt:rate_buy")
async def start_change_buy_rate(callback: CallbackQuery, state: FSMContext, session: AsyncSession, user: User | None) -> None:
    if user is None or user.role not in (RoleEnum.admin, RoleEnum.super_admin):
        await callback.answer("У вас нет прав.", show_alert=True)
        return
```

**RoleFilter** и **RoleGuardMiddleware** продолжают использоваться как дополнительный слой защиты, проверяя числовые уровни ролей (1 до 4).

## Сообщения при нарушении прав

| Ситуация | Сообщение |
|----------|-----------|
| Клиент пытается использовать команду оператора | «У вас нет прав для этого действия.» |
| Клиент обращается при отключённом боте | «Бот временно недоступен.» |
| Покупка при остановленном закупе | «Покупка USDT временно приостановлена.» |
| Продажа при остановленной продаже | «Продажа USDT временно приостановлена.» |
| Админ пытается назначить Админа | «У вас нет прав для этого действия.» |
| Назначение роли несуществующему пользователю | «Пользователь не найден. Он должен сначала запустить бота (/start).» |
