# Аудит безопасности проекта USDT Telegram Bot (Май 2026)

## Обзор
Был проведен комплексный аудит исходного кода и архитектуры проекта USDT Telegram Bot на предмет возможных уязвимостей, логических ошибок и проблем с отказоустойчивостью. Ниже представлены результаты аудита с категоризацией рисков.

---

## 🛑 КРИТИЧЕСКИЕ УЯЗВИМОСТИ (CRITICAL)

### 1. Полный обход авторизации (Broken Access Control / IDOR)
**Проблема:** В проекте реализован `RoleGuardMiddleware`, но он **абсолютно не функционирует**, так как обработчики (handlers) не передают параметр `required_role` в `data`. Более того, большинство обработчиков (callbacks и команды) вообще не имеют ручной проверки `user.role`.
**Последствия:** Любой обычный клиент (Client) может:
* Увидеть все активные заявки других людей, просто отправив боту текст `📋 Заявки`.
* Завершить любую чужую заявку, подделав callback `order_complete:<ID>` (в обработчике `complete_order.py` проверяется только наличие пользователя в БД, но не его права оператора).
* Отменить любую заявку через callback `order_admin_cancel:<ID>`.
* Изменить курсы валют (callback `mgmt:rate_buy` и `mgmt:rate_sell`), изменить реквизиты (callback `mgmt:links`) или чаты уведомлений (`mgmt:chats`), так как в `app/handlers/admin/management.py` отсутствуют проверки ролей на эти callback'и.

**Рекомендация:** Настроить передачу требуемых ролей в роутерах (например, через `router.message.filter(RoleFilter(min_role=...))`) или жестко добавить проверку `user.role` в **каждый** callback и message handler, относящийся к административной и операторской частям.

### 2. Отключенный антиспам (Throttling) — угроза отказа в обслуживании (DoS)
**Проблема:** В файле `app/bot.py` закомментировано подключение `ThrottlingMiddleware` с пометкой `DISABLED FOR DEBUGGING`.
**Последствия:** 
* Отсутствует защита от флуда. Злоумышленник может написать простой скрипт, отправляющий тысячи запросов в секунду.
* Это приведет к моментальному исчерпанию пула соединений PostgreSQL (Connection Pool Exhaustion).
* Telegram API выдаст блокировку `HTTP 429 Too Many Requests`, что полностью остановит работу бота для всех реальных клиентов.

**Рекомендация:** Немедленно раскомментировать и правильно настроить `ThrottlingMiddleware` для всех `message` и `callback_query`. При необходимости добавить Redis-хранилище для лимитов запросов (Rate Limiting).

## 🟠 ВЫСОКИЙ РИСК (HIGH)

### 3. Небезопасная аутентификация в REST API
**Проблема:** В `app/api/routers/auth.py` вход в API (`/api/v1/auth/login`) осуществляется путем проверки одного глобального `API_SECRET_KEY` и переданного `telegram_id` администратора.
**Последствия:** `API_SECRET_KEY` является статичным единым паролем для всего приложения. В случае его утечки, злоумышленнику достаточно знать (или подобрать) `telegram_id` любого оператора/администратора для получения полных JWT токенов (Access/Refresh) и захвата API.

**Рекомендация:** Вместо единого мастер-ключа использовать индивидуальные пароли для каждого администратора (сохранять их хэши в БД) или использовать авторизацию через Telegram Mini App (Web App Data / `initData`) с криптографической проверкой подписи от бота.

## 🟡 СРЕДНИЙ РИСК (MEDIUM)

### 4. Некорректная обработка состояния FSM (Finite State Machine)
**Проблема:** Механизм FSM позволяет любому пользователю находиться в состоянии `waiting_new_rate` (если он смог обойти интерфейс). В обработчиках вроде `process_new_buy_rate` в `change_rate.py` не проверяется роль пользователя перед сохранением нового курса в базу данных.
**Последствия:** Защита ложится исключительно на невозможность вызвать FSM. Если злоумышленник найдет способ активировать стейт (см. уязвимость №1), он беспрепятственно изменит финансовые настройки бота.

**Рекомендация:** Внутри каждого обработчика состояний `FSM` проверять роль (`if user.role != RoleEnum.super_admin: ...`).

### 5. Потенциальная утечка чувствительных данных через Global Error Handler
**Проблема:** При падении приложения (`app/bot.py -> global_error_handler`) используется `logger.exception("Unhandled exception in handler: %s", exception)`.
**Последствия:** Если в момент ошибки в переменных находились токены, приватные ключи шифрования `ENCRYPTION_KEY` или незашифрованные платежные реквизиты, они могут быть записаны в логи открытым текстом.

**Рекомендация:** Внедрить санитайзер логов, который будет маскировать чувствительные ключи (например, строки, похожие на `ENCRYPTION_KEY` или кредитные карты) перед отправкой в лог-файл или stdout.

## 🔵 НИЗКИЙ РИСК И АРХИТЕКТУРНЫЕ УЛУЧШЕНИЯ (LOW / INFO)

### 6. Вектор атаки на Worker (ARQ)
**Проблема:** Очередь Redis (`ARQ_REDIS_URL`) никак не сегментирована и работает в 1-й базе `redis://redis:6379/1` без пароля.
**Последствия:** Если внутренний контур Docker будет скомпрометирован (или порт Redis проброшен наружу), злоумышленник сможет инжектить произвольные задачи в ARQ (например, массовая рассылка фишинга от лица бота).
**Рекомендация:** Установить `REDIS_PASSWORD` в `docker-compose.yml` и использовать аутентифицированное подключение во всех сервисах.

### 7. Использование Long Polling для продакшена (в 2026 году)
**Проблема:** Бот использует метод `get_updates()` вместо Webhook. 
**Последствия:** Возможны потери сообщений при рестартах (хотя предусмотрен Graceful Shutdown), более высокое потребление ресурсов и задержки в обработке в моменты пиковых нагрузок.
**Рекомендация:** Запланировать переход на Webhooks поверх FastAPI / aiohttp, которые уже используются в проекте.

### 8. Отсутствие защиты от Replay Attacks в JWT
**Проблема:** Механизм обновления `refresh_token` не привязывает токен строго к Device Fingerprint или IP. 
**Рекомендация:** Привязать токены к IP-адресу или User-Agent'у и в случае их резкой смены инвалидировать текущую сессию.

---

**Итог:** Приложение находится в уязвимом состоянии из-за отключенных слоев защиты (RoleGuard, Throttling). Перед выводом в Production необходимо срочно исправить обходы авторизации в Callbacks и включить Anti-Spam Middleware.
